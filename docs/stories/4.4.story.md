## Story

**As the** system,
**I need** to enforce usage limits per tier so free users don't exceed their allocation and pro users get their monthly entitlement,
**so that** the billing model is sustainable and upgrade CTAs appear at the right moments.

## Acceptance Criteria

1. Free tier: 3 client max — blocked on attempt to create 4th client (UpgradeModal shown)
2. Free tier: 5 lifetime AI sessions max — blocked on attempt to trigger AI processing for 6th session
3. Free tier: 75 lifetime Solis queries max — blocked on 76th query
4. Pro tier: 25 AI sessions/month — monthly reset via `transcript_reset_at` timestamp
5. Pro tier: 2,000 Solis queries/month — monthly reset via `query_reset_at` timestamp
6. `usage_tracking` record auto-created (with 0 counts) on first limit check if not exists
7. Monthly reset logic: if `transcript_reset_at` is more than 30 days ago, reset `transcript_count = 0` and update `transcript_reset_at = now()`
8. UpgradeModal component displays with correct context (which limit was hit)
9. All limit checks happen server-side in API routes (client-side is only for UX)
10. 403 response returned with structured error when limit exceeded: `{ error: { code: 'LIMIT_EXCEEDED', message: '...', type: 'client' | 'transcript' | 'query' } }`

## Usage Tracking Table (already exists in migration 015)

```sql
-- DO NOT re-create — already in migration 015
-- user_id is UUID (not TEXT)
CREATE TABLE usage_tracking (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE UNIQUE,

  transcript_count INTEGER DEFAULT 0,
  transcript_reset_at TIMESTAMPTZ,  -- NULL = lifetime counter (free); set = last monthly reset (pro)

  query_count INTEGER DEFAULT 0,
  query_reset_at TIMESTAMPTZ,       -- NULL = lifetime counter (free); set = last monthly reset (pro)

  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

## Dev Notes

- Error class lives in `packages/shared/src/types/errors.ts` (not in app)
- API routes catch `UpgradeRequiredError` and return 403 with `{ error: { code: 'LIMIT_EXCEEDED', message, type: limitType } }`
- Frontend: check for 403 + `code: 'LIMIT_EXCEEDED'` → show UpgradeModal with `limitType` from response
- Monthly reset: compare `transcript_reset_at` to `new Date()` — if > 30 days, reset counter to 0 and update timestamp before checking
- Pro users get reset on upgrade (set `transcript_reset_at = now()`, `query_reset_at = now()`) — Story 5.1 handles this
- `getOrCreateUsageTracking`: use Supabase upsert with `onConflict: 'user_id'` to avoid race conditions
- Free users: `transcript_reset_at` and `query_reset_at` remain NULL (no reset ever)

## Risk & Compatibility

- **Primary risk:** Breaking existing client creation flow (currently has inline check at limit=1)
- **Mitigation:** Replace inline check with `checkClientLimit`; update limit from 1 → 3; test client creation before + after
- **Rollback:** Revert `checkClientLimit` call to inline check in clients route
- **No breaking DB changes** — `usage_tracking` table already exists; no schema modifications needed
