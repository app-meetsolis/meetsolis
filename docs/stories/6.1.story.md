# Story 6.1: Google Calendar OAuth + Upcoming Meeting Sync

## Status
Ready for Review

## Locked Decisions (2026-05-11)
- **OAuth scopes:** `calendar.events.readonly` + `calendar.readonly`. No write access — MeetSolis never modifies calendar.
- **Sync cadence:** 15-min polling via Vercel cron. Window = next 24 hours. Webhooks deferred (Google Calendar push notifications add complexity without payoff at this scale).
- **Token storage:** encrypted at rest (app-layer AES-256-GCM). Stored in `user_calendar_tokens`, RLS-protected.
- **Token refresh:** auto-refresh when `expiry_at - now() <= 5 min`. Refresh failure → mark connection broken → re-auth prompt.
- **Tier gating:** Calendar connection available to BOTH Free + Pro. Auto-bot is Pro-only (Story 6.2). Free coaches still see upcoming sessions in dashboard — drives Coach Brief upgrade pressure.
- **Client matching:** auto-match by attendee email against `clients.email`. Manual match fallback via dropdown of active clients. No fuzzy name matching (false-positive risk too high).
- **Settings location:** Calendar connection lives in **Settings → Preferences tab** (Story 5.4 ships the tab). New "Integrations" section.
- **Disconnect behavior:** revoke token via Google `oauth2.revoke` endpoint + delete row from `user_calendar_tokens`. Keep `calendar_events` rows for history (Coach Brief 6.4 references them). Reconnect resumes syncing.
- **Key rotation:** DEFERRED. Single-key encryption sufficient at launch scale (<1K users). Add rotation post-PMF or on compliance trigger.
- **Google verification:** ACCEPT "unverified app" warning at launch. Submit verification application on deploy day (6–8 week process runs in parallel). Add disclosure in onboarding copy.
- **Cron auth:** Vercel auto-injected `CRON_SECRET` (Bearer header validation). No custom secret.
- **Disconnect data retention:** Keep `calendar_events` rows. Briefs reference event IDs — deletion breaks history. Account-level wipe deferred to Story 5.10.

## Story

**As a** coach (Free or Pro),
**I want** to connect my Google Calendar to MeetSolis,
**so that** my upcoming coaching sessions auto-appear in my dashboard and the Coach Brief (Story 6.4) has the data it needs to auto-activate.

## Acceptance Criteria

### OAuth Flow
- [ ] **"Connect Google Calendar" button** added to Settings → Preferences tab. New section header: "Integrations". Below existing email-notifications + timezone settings.
- [ ] Click → redirect to Google OAuth consent screen via `GET /api/calendar/connect` → user grants `calendar.events.readonly` + `calendar.readonly` scopes.
- [ ] OAuth callback at `GET /api/calendar/callback?code=...&state=...`:
  - Validate `state` param (CSRF protection — generated server-side, stored in HTTP-only cookie before redirect).
  - Exchange code for tokens via Google token endpoint.
  - Encrypt `access_token` + `refresh_token` before insert.
  - Upsert into `user_calendar_tokens` (one row per user).
  - Redirect back to `/settings/preferences` with success toast.
- [ ] **Token storage:** `user_calendar_tokens` columns: `id`, `user_id` (FK to `users`, unique), `access_token_encrypted`, `refresh_token_encrypted`, `expiry_at` (timestamptz), `google_account_email`, `created_at`, `updated_at`.
- [ ] **Token refresh:** helper `getValidAccessToken(userId)` — checks `expiry_at`, refreshes if within 5 min, updates row, returns plaintext token. Called by sync job.
- [ ] **Refresh failure** (refresh token revoked / invalid): set `user_calendar_tokens.connection_broken = true`, surface re-auth prompt in dashboard.
- [ ] **Disconnect button** in Integrations section: `POST /api/calendar/disconnect` → call Google `oauth2.revoke` with refresh token → delete row → toast → return to disconnected state.
- [ ] **Connection status UI** in Integrations section:
  - Connected: green dot + `"Connected: {google_account_email}"` + Disconnect button.
  - Not connected: "Connect Google Calendar" button.
  - Broken: red dot + "Reconnect" button + tooltip "Your Google Calendar connection has expired. Reconnect to resume syncing."
- [ ] **RLS on `user_calendar_tokens`:** users can SELECT/INSERT/UPDATE/DELETE only their own row.

### Meeting Sync
- [ ] **Vercel cron job** configured in `apps/web/vercel.json`: hits `POST /api/calendar/sync-all` every 15 minutes. Job iterates all users with valid `user_calendar_tokens` and syncs their upcoming events.
- [ ] **Per-user sync logic** (`apps/web/src/lib/services/calendar/google-calendar.ts`):
  - Call Google Calendar API `events.list` with `timeMin=now`, `timeMax=now+24h`, `singleEvents=true`, `orderBy=startTime`.
  - Filter events: include only those with **Google Meet link** OR **Zoom link in description/location** OR **attendee email matching a Client Card** for this user.
  - Upsert into `calendar_events` by `google_event_id` (unique).
- [ ] **`calendar_events` table** columns: `id` (uuid), `user_id` (FK), `google_event_id` (text, unique with user_id), `title`, `start_time`, `end_time`, `attendees` (jsonb array of email strings), `client_id` (uuid, nullable, FK to clients), `meet_link` (text, nullable), `bot_status` (text, nullable — populated by Story 6.2), `bot_skipped` (boolean, default false — populated by Story 6.5), `synced_at`, `created_at`.
- [ ] **Manual sync trigger:** `POST /api/calendar/sync` (per-user, called from dashboard) → triggers a single user's sync immediately. Used by "Refresh" button in Upcoming Sessions card.
- [ ] **RLS on `calendar_events`:** users only see their own events.

### Dashboard — Upcoming Sessions Card

- [ ] **Loading state:** skeleton with 3 rows.

### Client Matching
- [ ] **Auto-match on sync:** for each fetched event, compare every attendee email against `clients.email` for this user's active clients. First match wins. Store in `calendar_events.client_id`.
- [ ] **Manual match modal** (triggered from Upcoming Sessions card "Match to client" link):

  - Optional: "Remember this email for {Client Name}" checkbox → also updates `clients.email` if currently null.
- [ ] **Unmatch flow:** matched event row shows pencil icon → click → reopens match modal with current selection pre-filled → can change or clear.

## Technical Notes

### Recommended File Layout
```
apps/web/src/app/api/calendar/
  connect/route.ts         # GET — initiates OAuth (sets state cookie, redirects)
  callback/route.ts        # GET — exchanges code, stores tokens, redirects
  disconnect/route.ts      # POST — revokes + deletes
  sync/route.ts            # POST — per-user manual sync
  sync-all/route.ts        # POST — cron-triggered, iterates all users
  events/[id]/route.ts     # PATCH — set client_id (manual match)

apps/web/src/lib/services/calendar/
  google-calendar.ts       # API client wrapper
  token-encryption.ts      # AES-256-GCM helpers
  sync-user-events.ts      # per-user sync logic
  match-client.ts          # auto-match by email

apps/web/src/components/dashboard/
  UpcomingSessionsCard.tsx # new dashboard card
  MatchClientModal.tsx     # match-to-client modal

apps/web/src/components/settings/
  CalendarIntegration.tsx  # Preferences tab section
```

### Library Choices
- **`googleapis`** npm package — official Google API client. Handles OAuth + Calendar API. Adds ~200KB to server bundle (server-only, no impact on client bundle).
- **Token encryption:** Node built-in `crypto`. No external dep.
- **TanStack Query:** already in project. Use `useQuery` for fetching `/api/calendar/events?limit=3`.

### Cron Setup
`apps/web/vercel.json`:
```json
{
  "crons": [
    { "path": "/api/calendar/sync-all", "schedule": "*/15 * * * *" }
  ]
}
```
Sync-all route must authenticate via `CRON_SECRET` header (`Authorization: Bearer ${CRON_SECRET}`) — Vercel auto-injects this. Reject other callers.

### DB Migration
```sql
-- user_calendar_tokens
CREATE TABLE user_calendar_tokens (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL UNIQUE REFERENCES users(id) ON DELETE CASCADE,
  access_token_encrypted text NOT NULL,
  refresh_token_encrypted text NOT NULL,
  expiry_at timestamptz NOT NULL,
  google_account_email text NOT NULL,
  connection_broken boolean NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
ALTER TABLE user_calendar_tokens ENABLE ROW LEVEL SECURITY;
CREATE POLICY user_calendar_tokens_owner ON user_calendar_tokens
  FOR ALL USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid());

-- calendar_events
CREATE TABLE calendar_events (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  google_event_id text NOT NULL,
  title text NOT NULL,
  start_time timestamptz NOT NULL,
  end_time timestamptz NOT NULL,
  attendees jsonb NOT NULL DEFAULT '[]'::jsonb,
  client_id uuid REFERENCES clients(id) ON DELETE SET NULL,
  meet_link text,
  bot_status text,
  bot_skipped boolean NOT NULL DEFAULT false,
  synced_at timestamptz NOT NULL DEFAULT now(),
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, google_event_id)
);
CREATE INDEX calendar_events_user_start_idx ON calendar_events (user_id, start_time);
ALTER TABLE calendar_events ENABLE ROW LEVEL SECURITY;
CREATE POLICY calendar_events_owner ON calendar_events
  FOR ALL USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid());
```

Update `packages/shared/src/types/` — add `UserCalendarToken`, `CalendarEvent` types.

### New Env Vars
- `GOOGLE_CALENDAR_CLIENT_ID` — OAuth client ID from Google Cloud Console.
- `GOOGLE_CALENDAR_CLIENT_SECRET` — OAuth client secret.
- `GOOGLE_CALENDAR_REDIRECT_URI` — production callback URL.
- `CALENDAR_TOKEN_ENCRYPTION_KEY` — 32-byte base64 string (generate via `openssl rand -base64 32`).
- `CRON_SECRET` — Vercel-managed cron auth secret (auto-injected).

Add all to `.env.example` with placeholder values + comment.

### State Cookie for CSRF
`/api/calendar/connect` generates random `state`, stores in HTTP-only secure cookie (`__Host-calendar_state`, SameSite=Lax, 5-min expiry). Callback validates cookie value against `?state=` query param, deletes cookie after use.
