## Locked Decisions (2026-05-14)
- **Transcription provider:** Deepgram Nova-2 (streaming for live, async fallback unchanged). Same cost as async ($0.0043/min) ŌĆö zero COGS delta.
- **Streaming transport:** Recall.ai-managed WebSocket to Deepgram. Recall fires `transcript.data` webhook events to us; we never hold a WS open ourselves. Keeps us serverless-friendly on Vercel.
- **Chunk persistence:** Append-only into `sessions.transcript_chunks JSONB` array. No separate `transcript_chunks` table ŌĆö keeps reads cheap, avoids join on every poll.
- **Finals only (corrected 2026-05-15):** Recall has no `is_final` field ŌĆö interim vs final is the *event type*. Subscribe to `transcript.data` (finalized utterances) only, NOT `transcript.partial_data`. Dedup = drop duplicate finals (Recall retries webhooks Ōēż60├Ś), matched by `start_ms`.
- **UI delivery:** Polling, not WS/SSE. UI polls `/api/sessions/[id]/transcript-live` every 3s while `transcript_streaming_complete = false`. Cheaper than holding open connections; latency acceptable for coaching context.
- **Sessions row lifecycle:** Created on first `transcript.data` OR `bot.in_call_recording` (whichever fires first) ŌĆö not at bot dispatch. Keeps sessions table clean of aborted bots.
- **Post-call pipeline:** On `bot.call_ended` ŌåÆ concatenate chunks into `transcript_text` ŌåÆ trigger `runSummarize` directly. Skip `runTranscribe` entirely (we already have the transcript). Cuts post-call latency from 3ŌĆō5 min ŌåÆ ~20s.
- **Audio playback:** HTML5 `<audio>` + signed Recall.ai S3 URL. URLs expire ~24h ŌĆö refresh on demand via `GET /recording/{id}`.
- **Sync mechanism:** Click line ŌåÆ `audio.currentTime = chunk.start_ms / 1000`. Highlight current line via `timeupdate` event matching `start_ms <= currentTime <= end_ms`.
- **Phase gating:** Phases ship independently. Phase 1 = operator env config (zero code). Phase 2 = streaming. Phase 3 = audio playback. Phase 1 ships immediately; 2 + 3 ship together if Phase 2 lands cleanly.
- **Speaker labels (corrected 2026-05-15):** Recall delivers `participant.id` (integer) + `participant.name` (real name, often populated by Meet/Zoom). Store both ŌĆö `speaker: number` + `speaker_name: string | null`. Render name when present, else `Speaker {id}`. Coach/client identity mapping deferred to Story 6.3.
- **No partial summary mid-meeting:** AI summarization runs only after `bot.call_ended`. Streaming is transcript-only.

## Story

**As a** Pro coach,
**I want** the transcript to appear live on my dashboard while a coaching session is happening, and to play back the recorded audio with each line clickable to jump to that moment,
**so that** I can review breakthrough moments faster than scrubbing through video ŌĆö and never wait 5 minutes after the call to see what was said.

**Already shipped (Story 6.2):**
- Recall.ai bot auto-joins Meet/Zoom sessions when calendar event matches dispatch window.
- `recall_sessions` table tracks bot lifecycle (`pending ŌåÆ joining ŌåÆ in_meeting ŌåÆ done`).

## Open Questions / Risks

1. **Webhook ordering:** can `transcript.data` events arrive out of order? Recall docs unclear. Mitigation: dedup by `start_ms`, sort on read in API endpoint.
2. **Sessions row race:** two `transcript.data` events fire simultaneously before sessions row exists ŌåÆ both try INSERT ŌåÆ unique constraint conflict. Mitigation: `INSERT ... ON CONFLICT (recall_session_id) DO NOTHING` then UPDATE.
3. **Mid-meeting page load:** coach opens dashboard 10 min into a 30-min call ŌĆö does `LiveTranscriptPanel` correctly load chunks-so-far + start polling? Test path required.
4. **Audio URL expiry during playback:** user starts playback at minute 0, listens for >24h (unlikely but possible) ŌåÆ URL expires mid-stream. Mitigation: `<audio onerror>` triggers refresh + reload at same `currentTime`.
5. **Deepgram streaming quality vs async:** docs note ~5ŌĆō10% WER delta. Acceptable for live use but may surface in coach feedback. Monitor sentiment.
6. **Polling cost:** 3s polling ├Ś 60-min meeting ├Ś 100 concurrent coaches = 1.2M Vercel function invocations/mo. Stays under Vercel Pro included quota but monitor.
