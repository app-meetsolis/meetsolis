# Story 6.2b: Real-Time Transcription + Audio Playback

## Status
Backlog

## Locked Decisions (2026-05-14)
- **Transcription provider:** Deepgram Nova-2 (streaming for live, async fallback unchanged). Same cost as async ($0.0043/min) — zero COGS delta.
- **Streaming transport:** Recall.ai-managed WebSocket to Deepgram. Recall fires `transcript.data` webhook events to us; we never hold a WS open ourselves. Keeps us serverless-friendly on Vercel.
- **Chunk persistence:** Append-only into `sessions.transcript_chunks JSONB` array. No separate `transcript_chunks` table — keeps reads cheap, avoids join on every poll.
- **Dedup strategy:** Keep only `is_final: true` chunks. Interim chunks dropped on arrival of matching final (matched by `start_ms` window).
- **UI delivery:** Polling, not WS/SSE. UI polls `/api/sessions/[id]/transcript-live` every 3s while `transcript_streaming_complete = false`. Cheaper than holding open connections; latency acceptable for coaching context.
- **Sessions row lifecycle:** Created on first `transcript.data` OR `bot.in_call_recording` (whichever fires first) — not at bot dispatch. Keeps sessions table clean of aborted bots.
- **Post-call pipeline:** On `bot.call_ended` → concatenate chunks into `transcript_text` → trigger `runSummarize` directly. Skip `runTranscribe` entirely (we already have the transcript). Cuts post-call latency from 3–5 min → ~20s.
- **Audio playback:** HTML5 `<audio>` + signed Recall.ai S3 URL. URLs expire ~24h — refresh on demand via `GET /recording/{id}`.
- **Sync mechanism:** Click line → `audio.currentTime = chunk.start_ms / 1000`. Highlight current line via `timeupdate` event matching `start_ms <= currentTime <= end_ms`.
- **Phase gating:** Phases ship independently. Phase 1 = operator env config (zero code). Phase 2 = streaming. Phase 3 = audio playback. Phase 1 ships immediately; 2 + 3 ship together if Phase 2 lands cleanly.
- **No diarization remap:** Use Deepgram speaker labels as-is (`Speaker 0`, `Speaker 1`). Coach/client identity mapping deferred to Story 6.3.
- **No partial summary mid-meeting:** AI summarization runs only after `bot.call_ended`. Streaming is transcript-only.

## Story

**As a** Pro coach,
**I want** the transcript to appear live on my dashboard while a coaching session is happening, and to play back the recorded audio with each line clickable to jump to that moment,
**so that** I can review breakthrough moments faster than scrubbing through video — and never wait 5 minutes after the call to see what was said.

## Context

**Already shipped (Story 6.2):**
- Recall.ai bot auto-joins Meet/Zoom sessions when calendar event matches dispatch window.
- `recall_sessions` table tracks bot lifecycle (`pending → joining → in_meeting → done`).
- Webhook handler at `/api/recall/webhook` processes `bot.joining_call`, `bot.in_call_recording`, `bot.call_ended`, `bot.done`, `bot.fatal`.
- Post-call: `raw_recording_url` saved; downstream transcription handled by existing async pipeline (Story 3.3) — which **currently falls back to MOCK data** because `DEEPGRAM_API_KEY` + `ANTHROPIC_API_KEY` not set in prod.

**What's new:**
- **Phase 1 (ops):** Set 4 env vars in Vercel → mock transcripts replaced with real ones. Zero code.
- **Phase 2 (dev):** Streaming transcription via Recall + Deepgram streaming provider. Chunks land in DB during meeting. Live UI panel polls and renders. Summarization fires on call end.
- **Phase 3 (dev):** Audio player + clickable synced transcript on session detail page.

## Open Questions / Risks

1. **Webhook ordering:** can `transcript.data` events arrive out of order? Recall docs unclear. Mitigation: dedup by `start_ms`, sort on read in API endpoint.
2. **Sessions row race:** two `transcript.data` events fire simultaneously before sessions row exists → both try INSERT → unique constraint conflict. Mitigation: `INSERT ... ON CONFLICT (recall_session_id) DO NOTHING` then UPDATE.
3. **Mid-meeting page load:** coach opens dashboard 10 min into a 30-min call — does `LiveTranscriptPanel` correctly load chunks-so-far + start polling? Test path required.
4. **Audio URL expiry during playback:** user starts playback at minute 0, listens for >24h (unlikely but possible) → URL expires mid-stream. Mitigation: `<audio onerror>` triggers refresh + reload at same `currentTime`.
5. **Deepgram streaming quality vs async:** docs note ~5–10% WER delta. Acceptable for live use but may surface in coach feedback. Monitor sentiment.
6. **Polling cost:** 3s polling × 60-min meeting × 100 concurrent coaches = 1.2M Vercel function invocations/mo. Stays under Vercel Pro included quota but monitor.
