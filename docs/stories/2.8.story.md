# Story 2.8: Client Deletion & Cascading

## Status
Done

## Story

**As a** executive coach,
**I want** to permanently delete a client and all their associated data,
**so that** I can remove clients who are no longer active and keep my workspace clean.

## Story Context

**Existing System Integration:**
- Integrates with: `DELETE /api/clients/[id]` (Story 2.1, fully implemented)
- DB cascade already in place (migration 015): `sessions` ON DELETE CASCADE, `action_items` ON DELETE CASCADE, `solis_queries.client_id` ON DELETE SET NULL
- Delete trigger placement: `ClientCard.tsx` via 3-dot menu (Story 2.6 client detail page not yet built)
- Technology: Next.js App Router, Supabase, React Query (`@tanstack/react-query`), Shadcn UI Dialog, `sonner` toast, Lucide icons, TypeScript
- Follows pattern: `ClientModal.tsx` for dialog pattern; `clients/page.tsx` for `useQueryClient` + `queryClient.invalidateQueries`
- Touch points: `ClientCard.tsx` (add delete trigger), new `DeleteClientDialog.tsx`, `api/clients/[id]/route.ts` (add Storage cleanup)

**Key pre-existing facts:**
- `DELETE /api/clients/[id]` returns `204 No Content` on success — do NOT change the response shape
- DB cascade handles sessions, action_items automatically — no manual cascade needed in API
- `solis_queries.client_id` is SET NULL (not CASCADE) — Solis history preserved but orphaned
- Sessions feature not built yet (Story 3.2/3.3) so Supabase Storage files won't exist at runtime, but add cleanup code defensively for when they do
- Correct redirect target: `/clients` (NOT `/dashboard/clients` — route group `(dashboard)` has no URL prefix)
- Toast library: `sonner` (already imported in `clients/page.tsx`) — NOT `react-toastify`

## Acceptance Criteria

1. 3-dot menu (or delete icon button) on `ClientCard` triggers delete flow
2. `DeleteClientDialog` opens with warning: "Delete [client name]? This will permanently delete all sessions, action items, and Solis query history. This cannot be undone."
3. Checkbox "I understand this is permanent" must be checked before Delete button is enabled
4. Delete button shows loading spinner during deletion; disabled to prevent double-submit
5. On confirm: `DELETE /api/clients/[id]` called
6. Cascade deletion removes all related records via DB constraints (sessions, action_items automatically; solis_queries nulled)
7. `DELETE /api/clients/[id]` also attempts Supabase Storage cleanup for `transcripts/{userId}/{clientId}/` path (defensive — no files exist until Story 3.2)
8. On success: redirect to `/clients`, toast "Client deleted"
9. On error: toast "Failed to delete client. Please try again.", client not deleted, dialog stays open
10. React Query clients cache invalidated on success
11. Existing client CRUD flow unaffected (no regression)

## Dev Notes

**Component architecture decision:**
Dialog state can live in `clients/page.tsx` (same as `ClientModal` pattern) OR in `ClientCard` directly. Recommend page-level state for consistency with existing `ClientModal` pattern — avoids prop drilling complexity in card.

**Delete flow (sequence):**
```
User clicks Delete in card menu
  → DeleteClientDialog opens (isOpen=true, client=selectedClient)
  → User checks checkbox
  → User clicks Delete button
  → isDeleting = true
  → fetch DELETE /api/clients/[id]
  → On 204:
      queryClient.invalidateQueries({ queryKey: ['clients'] })
      toast.success('Client deleted')
      router.push('/clients')
      onClose()
  → On error:
      toast.error('Failed to delete client. Please try again.')
      isDeleting = false
```

**API Storage cleanup (defensive):**
```typescript
// After DB delete succeeds — non-fatal
try {
  const { data: files } = await supabase.storage
    .from('transcripts')
    .list(`${userId}/${clientId}`);
  if (files && files.length > 0) {
    const paths = files.map(f => `${userId}/${clientId}/${f.name}`);
    await supabase.storage.from('transcripts').remove(paths);
  }
} catch (storageErr) {
  console.warn('[Clients API] Storage cleanup failed (non-fatal):', storageErr);
}
```
