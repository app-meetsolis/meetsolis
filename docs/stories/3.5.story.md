# Story 3.5: Session Timeline UI

## Status

## Story

**As a** executive coach,
**I want** to see all my sessions for a client in a rich, interactive timeline,
**so that** I can quickly review history, drill into summaries, and upload new sessions.

## Acceptance Criteria

**Functional:**

1. `SessionTimeline` replaces `SessionTimelineStub` on client detail page ‚Äî no visible regression for existing data
2. Sessions displayed in reverse-chronological order (newest first)
3. "Upload Session Transcript" button at top opens existing `SessionUploadModal`
4. Empty state: "No sessions yet. Upload your first session transcript to get started." + upload button
5. Loading skeleton: 3 skeleton cards (~80px height each) while fetching
6. **SessionCard collapsed:** formatted date (`Feb 15, 2026`), title, summary snippet (first 150 chars of `summary`), key topic badges (max 3, `‚Ä¶` if more), action item count badge ("3 actions" ‚Äî green if all complete, amber if any pending) ‚Äî badge derived from `action_items` array in API response (no extra per-card requests)
7. **SessionCard expanded (click header to toggle):** full summary, all `key_topics` as badges, `ActionItemList` placeholder (renders `null` for now ‚Äî Story 3.6 fills it), "View Full Transcript" link (if `transcript_text` or `transcript_file_url` exists) ‚Äî navigates to `/clients/[clientId]/sessions/[sessionId]`
8. Sessions with `status = 'processing'` show animated spinner + "AI is processing your session‚Ä¶" text; no expand toggle
9. Sessions with `status = 'error'` show red border, error message, "Retry" button ‚Üí `POST /api/sessions/[id]/summarize`
10. React Query `refetchInterval`: 5s while any session has `status = 'processing'`, otherwise disabled
11. Mobile responsive: cards stack vertically, expand/collapse works on touch
12. `GET /api/sessions` response includes nested `action_items(id, status)` per session (via Supabase nested select)
13. Session detail stub page at `/clients/[clientId]/sessions/[sessionId]` renders: title, formatted date, scrollable transcript text ‚Äî read-only, back link to `/clients/[clientId]`

**Integration:**

14. `SessionUploadModal` reused as-is (no changes to it)
15. On upload success ‚Üí `queryClient.invalidateQueries(['sessions', clientId])`
16. Sessions query cache key `['sessions', clientId]` unchanged ‚Äî no double-fetch with stub

**Quality:**

17. No regression on client detail page (existing stats, notes, etc.)
18. All new files ‚â§500 lines
19. No `process.env` direct access

## Dev Notes

- `SessionUploadModal` already exists at `apps/web/src/components/sessions/SessionUploadModal.tsx` ‚Äî import as-is
- Date format: use `date-fns` `format(parseISO(session.session_date), 'MMM d, yyyy')`
- Topic badges: shadcn/ui `Badge` with `variant="secondary"`
- Action count badge: green (`bg-green-100 text-green-700`) if all complete, amber (`bg-amber-100 text-amber-700`) if any pending ‚Äî derive from `session.action_items` array, omit badge if array absent/empty
- Supabase nested select: `.select('*, action_items(id, status)')` returns `action_items` as an array on each session object ‚Äî no extra requests
- Processing spinner: Tailwind `animate-spin` on a `Loader2` lucide icon
- `ActionItemList` slot: render `null` with `{/* TODO: Story 3.6 */}` comment
- Transcript page is intentionally minimal ‚Äî no edit, no delete (Story 3.x+ will enrich)
- If no single-session GET route exists, add `GET /api/sessions/[id]/route.ts` as a micro-task (reuse auth + ownership pattern from summarize route)
