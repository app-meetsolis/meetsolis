## Story

**As a** executive coach,
**I want** an AI-generated summary of my coaching session with key topics and action items extracted automatically,
**so that** I spend zero time on post-session documentation.

## Acceptance Criteria

**Functional:**

1. `POST /api/sessions/[id]/summarize` calls real AI provider (controlled by `AI_PROVIDER` env var: `placeholder` | `openai` | `claude`). **Default: `openai`**
2. AI generates structured output: `title` (string), `summary` (paragraph), `key_topics` (string[]), `action_items` ({ description, assigned_to: 'coach' | 'client' }[])
3. Each extracted action item is created as an `action_items` record with correct `session_id`, `client_id`, `user_id`, `assigned_to`
4. `sessions.summary`, `sessions.key_topics` updated with AI output
5. `sessions.embedding` (vector 1536) populated via `text-embedding-3-small` (OpenAI) ‚Äî real embeddings, no stub
6. Session `status` ‚Üí `'complete'` on success, `'error'` on failure
7. Placeholder provider returns realistic mock coaching output instantly (no API cost) ‚Äî default for local dev
8. Manual edit of summary: `PUT /api/sessions/[id]` with `{ summary }` saves changes (route already exists; just verify it handles `summary` field)
9. "Regenerate" re-calls `POST /api/sessions/[id]/summarize` ‚Äî existing action items NOT deleted, new ones appended (or idempotent ‚Äî design choice, note in dev notes)

**Integration:**

10. Existing `POST /api/sessions` fire-and-forget summarize trigger (sessions/route.ts:337‚Äì341) continues to work unchanged
11. Auth + ownership checks in summarize route maintained (already present in stub)
12. `AI_PROVIDER` and API keys read from `config.ai` (not `process.env` directly)

**Quality:**

13. Prompt templates use ICF-aligned coaching vocabulary (no clinical/therapeutic language)
14. Malformed AI response handled gracefully: partial extraction or status ‚Üí `'error'` with descriptive message
15. Cost target for OpenAI: <$0.05/session using GPT-4o-mini. If `AI_PROVIDER=claude`, enable prompt caching (`cache_control: { type: 'ephemeral' }`) to hit same target.

### Refactoring Performed

- **File**: `apps/web/src/lib/sessions/summarize-session.ts`
  - **Change**: Changed return type from `Promise<void>` to `Promise<'complete' | 'error' | 'skipped'>` with explicit return values
  - **Why**: Route was always returning `200 { status: 'complete' }` regardless of AI outcome. If AI failed, DB would be set to `error` but HTTP caller would see success ‚Äî misleading for Story 3.5 Regenerate flow
  - **How**: Returns outcome string; route now returns 500 on error, 200 with actual status on success

- **File**: `apps/web/src/app/api/sessions/[id]/summarize/route.ts`
  - **Change**: Consume `runSummarize()` return value and propagate error as 500
  - **Why**: Same as above ‚Äî HTTP semantics should reflect actual outcome
  - **How**: `result === 'error'` ‚Üí return 500; otherwise return 200 with `{ status: result }`
