# Story 5.4: Settings Page (Profile, Billing, Preferences)

Ready for Review

## Locked Decisions (2026-05-09)
- **Profile mgmt:** Embed Clerk `<UserProfile />` (handles email change + verification, name, avatar, password, connected accounts, sessions). Themed via `appearance` to match brand.
- **IA:** Single `/settings` route with **Clerk-owned routing** (`<UserProfile />` renders Profile/Security/Sessions tabs natively; we add **Billing** and **Preferences** as custom `<UserProfile.Page>` children).
- **MFA:** Hidden in v1 (disable at Clerk instance level — Dashboard → User & Auth → Multi-factor).
- **Account deletion:** Out of scope. Deferred to Story 5.10 (full GDPR cascade flow).
- **Preferences tab content:** email-notifications toggle + timezone. **Theme toggle SPLIT OUT** to follow-up story (`next-themes` not installed; only 2 `.dark` selectors in globals.css → dark palette not comprehensively defined; audit + token migration is 2–3 days, doesn't fit 5.4 envelope).
- **Cancel CTA copy:** "Cancel subscription" (industry-standard — Netflix/Spotify/Vercel pattern). Confirmation dialog carries the "no surprise charge" message.
- **Cancelled-active Pro badge:** Show `"Pro · Cancels {date}"` text (not live countdown) + inline "Resume subscription" link (recovery path = ~30% reversal lift).
- **Timezone UX:** Curated ~50 common business timezones by default + "Show all timezones" expander that loads full `Intl.supportedValuesOf('timeZone')`.
- **`email_notifications_enabled` default:** `true` (opt-out). Transactional emails (receipts, payment-failed) always on regardless.
- **Cancellation backend:** Dodo Payments has no hosted customer portal → ship `POST /api/billing/cancel` server route that calls Dodo API.

## Story

**As a** signed-in user (free or Pro coach),
**I want** a single `/settings` page where I can manage my profile (email, name, avatar, password), see my plan + usage, cancel/upgrade, and tune notifications/theme/timezone,
**so that** I can stay in control of my account without contacting support or hunting across pages.

## Technical Notes

### Recommended File Layout

```
apps/web/src/app/(dashboard)/settings/
  [[...rest]]/
    page.tsx                # server wrapper; renders <SettingsClient />
  loading.tsx               # existing — keep
  components/
    SettingsClient.tsx      # "use client"; renders ClerkUserProfile + custom pages
    BillingTab.tsx          # custom UserProfile.Page body
    PreferencesTab.tsx      # custom UserProfile.Page body
    PlanCard.tsx
    UsageCard.tsx

```

```tsx
// SettingsClient.tsx (sketch)
'use client';
import { UserProfile as ClerkUserProfile } from '@clerk/nextjs';
import { CreditCard, Sliders } from 'lucide-react';
import { BillingTab } from './BillingTab';
import { PreferencesTab } from './PreferencesTab';
import { settingsAppearance } from './appearance';

export function SettingsClient() {
  return (
    <ClerkUserProfile path="/settings" routing="path" appearance={settingsAppearance}>
      <ClerkUserProfile.Page label="Billing" url="billing" labelIcon={<CreditCard className="h-4 w-4" />}>
        <BillingTab />
      </ClerkUserProfile.Page>
      <ClerkUserProfile.Page label="Preferences" url="preferences" labelIcon={<Sliders className="h-4 w-4" />}>
        <PreferencesTab />
      </ClerkUserProfile.Page>
    </ClerkUserProfile>
  );
}
```

### Email-change flow
Clerk handles entirely via `<ClerkUserProfile />`. Verification code is sent to the **new** email. No backend work required — but **manually QA the flow in browser** (don't trust the component blindly).

### `/api/billing/cancel` endpoint
- `POST /api/billing/cancel` (auth required)
- Look up `subscriptions.dodo_subscription_id` for current user.
- Call Dodo API: `PATCH /subscriptions/{id}` with `{ cancel_at_next_billing_date: true }` (verify exact field name against Dodo docs — defer to dev).
- Return `{ ok: true, period_end: <iso> }`.
- Webhook (`subscription.cancelled`) eventually flips `subscriptions.plan` to free at period end (already wired in Story 5.1).

### Theme implementation — DEFERRED
Split out to a separate follow-up story (proposed: Story 5.14 "Dark/Light Theme System"). Reason: `next-themes` not installed; only 2 `.dark` selectors in `globals.css` despite 135 CSS vars → dark palette not comprehensively defined. Implementation requires audit + token migration across all dashboard components (est. 2–3 days), which doesn't fit 5.4's 1-day envelope. **Do not partially implement in this story.**

### DB migration
New columns on `users` table:
```sql
ALTER TABLE users
  ADD COLUMN IF NOT EXISTS email_notifications_enabled BOOLEAN NOT NULL DEFAULT true,
  ADD COLUMN IF NOT EXISTS timezone TEXT;
-- email_notifications_enabled defaults to true (opt-out model, see locked decisions)
```
Update `User` type in `packages/shared` accordingly. RLS: users can SELECT/UPDATE only their own row (existing policy should already cover this — verify).

### API routes added
- `POST /api/billing/cancel` — see above.
- `PATCH /api/user/preferences` — body `{ email_notifications_enabled?: boolean, timezone?: string }`. Auth required, Zod-validated.
